01 / 心智模型

Godot 是什么，GDScript 又是什么

Godot = 一个开源游戏引擎；GDScript = 它内建的脚本语言，语法像 Python，但为游戏和引擎类型系统量身定做。

两句话记住它们的关系：引擎负责渲染、物理、音频、输入、场景树这些"跑起来"的底座；你写的 GDScript 挂在场景里的节点上，每帧被引擎回调，用来表达游戏逻辑。你不写主循环——引擎替你循环，你只填 _ready() / _process(delta) 这些"钩子"。

引擎（C++ 内核）

渲染管线、物理、场景树调度、资源导入。你不碰它的源码，只调用它暴露的类：Node3D、Sprite3D、ShaderMaterial…

GDScript（你写的）

动态偏静态、带可选类型标注、缩进成块。热重载快，和引擎类型无缝互通。maliang 全部逻辑都是它。

场景 .tscn（数据）

节点树的序列化文本。谁是谁的孩子、挂什么脚本、初始属性，全在这里。可视化编辑器改的就是它。

着色器 .gdshader

跑在 GPU 上的一小段程序，决定每个顶点/像素怎么画。maliang 的"小星球弯曲"就是一个顶点着色器。

maliang 是什么？ 一个给不识字的幼儿园孩子玩的 Android 平板游戏：动漫风草原小村庄，点角色 → 语音对话 → 地图中央的"小神仙"能按孩子的口头想法实时用 AI 造出新角色。技术上是 HD-2D：3D 场景里立着 2D 纸片角色（织梦岛/纸片马里奥那种观感）。客户端是 Godot 4 + GDScript，后端是 Node/TypeScript。

这一个文件里出现的每样东西：

语法含义
class_name WorldGrid给这个脚本注册一个全局类名，别处直接 WorldGrid.wrap_pos(...) 就能用，不必先 load。
extends RefCounted继承。RefCounted 是引用计数对象（无需手动释放），适合"纯工具类"——它不是场景里的节点。

Vector2 / Vector2i引擎内建类型：二维浮点向量 / 二维整数向量。游戏里到处是它们。
fposmod(a, b)引擎全局数学函数（浮点取模，结果恒非负）——GDScript 自带一大批 clamp/lerp/posmod 之类。

03 / 引擎模型

节点、场景、场景树

Godot 的世界观只有一句话：一切都是节点（Node），节点组成树，树就是你的游戏。

一个"节点"是一个有具体职责的对象：Sprite3D 画一张 3D 空间里的图，Camera3D 是相机，AudioStreamPlayer 放声音，Timer 计时。你把它们拼成父子树，父节点移动/隐藏时孩子跟着走。一棵这样的树存成一个 .tscn 文件，就叫一个场景。

maliang 的主场景其实很朴素，看它的 .tscn 就懂了这个格式：

  • [ext_resource]：外部引用——这里引了脚本 menu.gd。
  • [node name="Menu" type="Control"]：一个 UI 根节点，类型 Control（所有 UI 控件的基类）。
  • script = ExtResource("1")：把上面那个脚本挂到这个节点上——脚本 = 节点的行为。

脚本第一行决定"我是哪种节点"

每个 .gd 顶部的 extends 说明它要挂在哪类节点上。这是理解任意 GDScript 文件的第一个动作——先看它 extends 什么：

Node3D

3D 空间里有位置/旋转/缩放的东西。maliang 的世界根、区块管理器都是它。

Sprite3D

在 3D 空间里画一张 2D 图片——正是 HD-2D 纸片角色的载体。

Node

最朴素的节点，没有空间概念。适合放"管理器/服务"，比如网络 Backend。

RefCounted

不进场景树的纯逻辑对象。WorldGrid/Pathfinder/Mover 全是它——算法归算法，不占节点。

读代码技巧：注意这些属性名（billboard、alpha_cut、pixel_size、offset）全是 Sprite3D 这个引擎类的公开属性。学 Godot 很大一部分就是"认识某个节点类有哪些属性可调"——编辑器右侧的 Inspector 面板列的就是它们，代码里赋值等价于在面板里拖。
04 / 通信与节奏

信号、生命周期回调

节点之间不靠互相持有引用乱调，而是发信号（signal）——观察者模式的引擎级实现。谁关心，谁自己去 connect。

声明信号

网络层 Backend 把"收到了什么"抽象成一组信号，自己完全不关心谁在听：

连接信号

世界场景 world.gd 在启动时把这些信号接到自己的处理函数上。语法是 信号.connect(处理函数)：

  • backend.character_response.connect(_on_character_response)：后端一旦发 character_response，引擎就替你调 _on_character_response(data)。
  • 信号可带参数（character_response(data: Dictionary)），参数原样传给处理函数。
  • UI 按钮也发信号：listen_btn.pressed.connect(_on_listen)——"聆听"按钮被按下就录音。
  • 还能连匿名函数：tts_end.connect(func() -> void: pass)。
为什么值得这么做？ Backend 不 import world，world 也不改 Backend 的代码——两者只通过信号名约定耦合。要加个新监听者（比如做个调试面板），再 connect 一次即可，谁都不用改。这就是解耦。

生命周期钩子：引擎替你循环

你几乎从不写 while true 主循环。引擎在固定时机回调这些以下划线开头的方法，你只填内容：

钩子何时被调典型用途
_ready()节点进入场景树、孩子都就绪后，调一次初始化：接信号、建子节点、拉数据
_process(delta)每一帧调一次，delta=距上帧秒数动画、跟随、逐帧推进状态机
_unhandled_input(event)有输入且没被 UI 吃掉时点地图移动、点角色交互
_init()对象刚构造时（连节点树都还没进）纯逻辑/测试脚本的入口
delta 是帧率无关的关键。 移动写成 位置 += 速度 * delta，无论 30fps 还是 120fps，每秒走的距离都一样。行为执行器里 step_vec = 方向 * SPEED * delta 正是这个道理——下一节会看到它的完整状态机。
05 / 工程

项目架构：薄客户端 + 全能后端

一条铁律贯穿全项目：客户端只做渲染 + 输入 + 缓存，所有复杂度与密钥都在后端。 APK 里绝不出现任何 API key。

┌──────────── Android 客户端 (Godot 4 · GDScript) ────────────┐ │ 渲染 HD-2D：Sprite3D 纸片 + world-bending 弯曲着色器 │ │ 逻辑 toroidal grid 环面世界 + chunk streaming 区块流送 │ │ 输入 点角色 → 交互模式 → 录音 → 发送 │ │ 缓存 已生成 sprite / 世界快照 本地缓存 │ └───────────────────────────┬─────────────────────────────────┘ │ WebSocket(流式/推送) + REST(存取/资源) ┌───────────────────────────┴─────────────────────────────────┐ │ 后端代理 (Node/TypeScript · 自托管 muvee) │ │ · 造角色编排：LLM → 生图 → 抠图 → 切片 → 审核 → 落地 │ │ · 长任务队列 + 进度推送（requestId + progress 阶段） │ │ · 世界状态持久化（角色 memory / chat / 位置） │ │ · ASR / TTS 代理 —— 所有 API key 只在这里 │ └──┬─────────┬─────────┬─────────┬─────────┬────────────────────┘ │ LLM │ 生图 │ ASR │ TTS │ 内容审核 Claude OpenRouter 讯飞 讯飞 文 + 图 双审

为什么这样切？三个工程理由：

安全

面向幼儿，内容必须审核；密钥不能进用户手机。把第三方调用与审核全锁在后端是唯一安全的切法。

延迟掩盖

造角色是 10–30s 长任务。后端用"requestId + 逐阶段进度推送"，客户端放"施法中"动画盖住等待。

可迭代

换 LLM、换生图模型、调审核策略，都只动后端，不用重新发 APK。

客户端脚本的分层

仓库里的 scripts/*.gd 大致分三层，看 extends 就能归类：

层extends代表文件职责
纯算法（无状态）RefCountedworld_grid · pathfinder · mover · autotile坐标数学、寻路、移动规则、拼接——可单测
场景节点Node3D / Sprite3Dworld · chunk_manager · paper_character渲染、每帧驱动、放置角色
服务/管理器Nodebackend · fairy_voice网络、语音，靠信号对外通信
这个分层的价值：算法层不碰引擎、不进场景树，于是能用 --headless 无窗口跑单元测试（第 8 节）。渲染层再复杂也只是"调用算法层 + 摆节点"。职责一分开，测试和调试都简单了。
06 / 算法

四个真正有意思的算法

这个项目工程含量最高的部分。四个算法，每个都能独立成篇。

6.1 环面世界（Toroidal Grid）：有限地图，无限循环

世界是 75×75 格的有限网格，但孩子往任意方向一直走都不会到"边缘"——因为它首尾相接，像《吃豆人》穿墙那样，拓扑上是一个环面（甜甜圈面）。实现只靠两个数学函数：

  • wrap_pos：任何坐标用 fposmod 折回 [0, 150)。走过右边界 149.9 → 0.1，无缝。
  • shortest_delta：从 A 到 B 的最短环面位移。关键技巧是 fposmod(dx + half, span) - half，把差值折进 [-75, +75)——于是从 999.x 到 0.x 算出的是 +10，而不是绕大半圈的 -140。
浮动原点渲染：玩家永远画在渲染原点 (0,0)，世界相对他滚动。每帧对每个物体用 shortest_delta 求"离玩家最近的等价位置"再摆放。这样跨 1000 单位边界时画面零跳变——这是无缝大世界的经典手法。

6.2 区块流送（Chunk Streaming）：对象池

不可能把整张地图的节点都实例化。chunk_manager.gd 只维护玩家周围 3×3 个区块的节点，用对象池复用：玩家移动时，滑出视野的区块不销毁，而是"搬"到新出现的位置并按 wrap 后的索引重刷外观。

为什么用池而不是"销毁+新建"？ 创建/释放节点有开销且产生 GC 抖动，在手机上尤其致命。固定 9 个 slot 反复搬位置，内存和帧时间都恒定。注意 CULL_MARGIN 那条注释——因为弯曲是 GPU 干的，CPU 端的视锥裁剪不知道顶点被压弯了，得给大裁剪边距防止整块地面被误剔除。这种"GPU 变形 vs CPU 裁剪"的错位是图形工程的典型坑。

6.3 半格 A* 寻路：带穿角防护与路径拉直

角色要绕开树、房子、彼此走到目标。pathfinder.gd 是一套手写 A*，且为环面世界和 HD-2D 做了几处精心处理。先看主循环：

四个值得学的设计点：

手写二叉堆

GDScript 没有内建优先队列，于是自己用数组实现最小堆（_heap_push/_heap_pop）。元素是 [f, 序号, cell]，序号做稳定 tiebreak，保证同代价路径确定。

环面 octile 启发

启发函数每个轴取最短 wrap 距离（min(d, CELLS-d)），否则跨接缝的目标会被高估、算出绕远路。

防穿角

对角走一步，要求两个正交邻居也可通行——否则角色会从两块石头的对角缝里"钻"过去。

路径拉直

A* 出的是 8 向锯齿。_smooth 做 string-pulling：从锚点贪心跳到最远可通视的 waypoint，把楼梯折线拉成自然直线。

一致性是灵魂：寻路的单步判定 _step_ok 和运行时移动的 Mover._passable 用完全相同的规则（地形台阶 + 双层占用）。否则会出现"寻路说能走，实际卡住"的鬼畜。连拉直后的直线段也按 0.1m 采样逐点复刻同一判定——保证规划与执行永不打架。

6.4 Blob 自动拼接（Autotile）：位掩码 → 边角变体

草地边缘要自动画出圆角过渡，不能每块手摆。经典做法看 8 个邻居生成一个位掩码，再查表。maliang 用更省的角/对偶网格变体：把一格拆成 4 个角，每个角只看 3 位邻居，得 5 种变体——比整块 47 变体简单得多，视觉等价。

  • const N := 1, NE := 2, E := 4 …：8 个方向各占一个二进制位，用 |= 拼成一个 0–255 的掩码。
  • mask & N != 0：位与判断"北边是不是同类"。
  • 每个角只喂 3 个 bool（横向、纵向、对角邻居），corner_variant 映射到 5 种角形。
  • 邻居判定用 Callable 外置（第 2 节那段），所以草地和悬崖复用同一套逻辑。

6.5 行为执行器：一台跑在每帧里的状态机

后端下发 { commands:[…], loop:true }，客户端逐条执行驱动角色移动。behavior_executor.gd 是一台 idle → move → wait → done 的状态机，step(delta) 每帧推进一次：

  • 成员变量存状态（_state / _idx / _waypoints）——它 extends RefCounted，一个执行器管一个角色。
  • 进 move 就调 6.3 的寻路拿 waypoint 队列；无路可走则回退直线滑动（_direct）。
  • 被别的角色挡住 → 原地等一拍再局部重算（REPATH_WAIT）；反复被挡超过 REPATH_MAX 次就放弃当前指令，防止"磨墙"。
  • deliver_message 是 maliang 的扩展：走到目标角色旁边，触发一次回调把话传到。
07 / GPU

着色器：把平地弯成小星球

maliang 那种"站在小行星上、远处地平线向下弯"的观感，不是建模建出来的——数据层永远是纯平的环面，弯曲只发生在 GPU 渲染的一瞬间。

整个魔法就一行顶点位移。着色器语言（GLSL 风格的 .gdshader）和 GDScript 是两回事，但很好读：

  • shader_type spatial：这是 3D 空间着色器（对应 2D 是 canvas_item）。
  • uniform：从 GDScript 传进来的参数。BendMat.make()（第 2 节工厂）就是在设 curvature / albedo。
  • void vertex()：对每个顶点跑一次。VERTEX.y -= curvature * dot(VERTEX.xz, VERTEX.xz)——离原点水平距离的平方决定下沉多少，于是近处几乎平、远处急弯，弯出弧形地平线。
  • void fragment()：对每个像素跑一次，决定颜色/粗糙度/法线。
注释里藏着一个真实教训：代码写 world_vertex_coords（世界空间位移）而不是视图空间。因为阴影是从光源视角再渲一遍的——若在"视图空间"弯曲，相机 pass 和阴影 pass 的"视图"不是同一个，阴影就会错位。改到世界空间后两个 pass 几何一致，阴影才对。这类"为什么非得这么写"的注释，是读工程代码时最该停下来品的地方。
