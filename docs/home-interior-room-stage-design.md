# 室内系统重做：房间舞台（Room Stage）设计方案

> 状态：待老板审。审过再进 plan 模式出 P 计划开建。
> 背景：初版把室内塞进无限连续 chunk 的开放地形（抬高地形块假装墙、掏实心躲露地），
> 方向错了——地形系统是为「无限/滚动/处处是地」的室外做的，室内要的是**有界的舞台**。
> 本方案换掉那个 hack，给室内做一个正经的房间舞台。

## 1. 目标观感（对齐动物森友会 / Pokopia 室内）

参考里的室内共性：
- **有界房间**（不是开放世界的一块）：固定尺寸的一间屋，相机框在这一间里。
- **真墙面 + 壁纸**：竖直墙面贴壁纸（可带花纹，甚至 accent 主题墙一面不同），底部有踢脚线。
- **地板花纹**：与墙面分开的地面材质（木地板/地毯/瓷砖…），地毯可圈定区域。
- **天花板**：v2 起动森有天花板；至少要有「头顶不是天、是收束的顶」。
- **相机是收束/固定的舞台视角**：不自由滚动、不见开放地形、不见天空。
- **暖房氛围**：室内暖光，安定、温馨。

参考来源（design 结构，非逐图）：Nookipedia「Designing」（AC 墙纸/地板/accent 墙/天花板）、
Pokopia 家具目录（Chairs/Tables/Beds/Storage/Decor…，摆够 3 件才入住）。

## 2. 核心判断：室内是「舞台」，不是「地形场景」

地形系统（WorldGrid 无限环面 + chunk_manager 滚动铺 chunk + 处处地面 + 天空 + 深度雾）
天生服务室外。室内塞进去必然要 hack（抬地形当墙、掏实心躲露地、BG 假装收顶），且：
- 墙是「抬高的地形块」，顶面是 tile 纹理 —— 永远不像壁纸墙面；
- chunk 处处铺地 —— 墙外/前景总会露地，只能靠遮挡硬躲；
- 相机自由滚动 —— 没有「一间屋」的框。

**结论**：进室内场景时，客户端**不铺地形 chunk**，改渲染一个**房间舞台节点**（真几何 + 收束相机）。

## 3. 房间舞台架构

进入 `home_interior`（及未来其它室内场景）时，客户端按「场景是室内」分支，构建一个 `RoomStage` 节点：

**几何（真 MeshInstance3D，非地形）：**
- **地板**：一块 `PlaneMesh`，铺地板材质（木地板/地毯，程序化或贴图）。尺寸 = 房间逻辑网格（拟 8×8 或 10×10 tile，够摆家具、又不空旷；比初版 16×16 更聚拢）。
- **墙面**：3 面竖直墙 quad（后墙 + 左右墙；朝相机的前墙留开口/不建，好看进屋内）——贴**壁纸材质** + 底部踢脚线；一面可做 accent 墙（后续）。
- **天花板**：一块顶 quad（暖色/木纹）或先留「收束暗顶」（不建 mesh、BG 暗色兜底），二选一由眼验定。
- **门**：后续做一扇门 prop（进/出的锚点），MVP 可先用墙上一处开口 + portal。

**相机（室内模式，顺 `_update_camera` 既有相机模式分支加）：**
- 现有相机 = 固定原点上方、靠 `focus_logical` 滚动世界（浮动原点），且已有 photo/creation/phone/dialog 多种模式覆盖。
- 加 `_room_cam` 模式：**收束的舞台视角**——固定俯角、把整间屋框进画面（房间小，可整屋入镜或轻微跟随玩家但夹在房间边界内），不自由滚动到屋外。参数（pitch/dist/lift）按房间尺寸反算，进室内切入、出屋复位。

**光照：** 室内暖光（复用 P3 已做的 per-scene 环境切换：无天空 BG_COLOR + 暖 ambient + 压暗太阳；可再加一盏室内 OmniLight 提亮中心）。

**渲染分支：** `_apply_scene`（或其上游）判断场景是否室内 → 室内则跳过 `chunk_manager` 铺地形、build `RoomStage`；室外维持现状。出室内 free 掉 RoomStage、复位相机、恢复地形。

## 4. 复用 vs 重做（重要：大半已验过的活能留）

| 模块 | 处置 |
|---|---|
| 走进家门进屋（村里 portal→home_interior，P2） | **复用**（实机验过）——portal 机制与房间怎么渲染无关 |
| 家具起始包发到背包（P4） | **复用**（实机验过） |
| 背包→选家具→布置模式→落地→持久化（P5） | **复用**，但布置的「落点网格」要映射到房间地板网格（见 §5） |
| 无天空/暖光/太阳压暗的室内环境（P3） | **复用**（per-scene 环境切换保留，是房间氛围的一部分） |
| 地形抬高块当四壁 + 掏实心（P1 `_paint_home_interior`） | **换掉**——由 RoomStage 真墙面/地板/天花板几何取代 |
| home_interior 的 .mltr 地形数据 | **简化/退役**——房间由客户端 RoomStage 定义；服务端场景仅存房间尺寸 + 家具放置（overlay） |

## 5. 家具放置如何落到房间

布置模式（幽灵选位/合法高亮/朝向/落地）与 OccupancyMap、物品层现在都挂在地形网格上。房间舞台方案下二选一：
- **A 房间仍保留一小块「逻辑网格」**（不渲染成地形，只做放置/占用坐标系）：布置照旧走 tile→落点，RoomStage 地板 mesh 与这块网格对齐。改动最小（放置链几乎零改），是**推荐**。
- B 放置完全 room-local 重写：更纯但重写放置，性价比低。

采 A：房间 = 一块 N×N 逻辑网格（放置/占用/家具 item 层都在其上），RoomStage 只负责把这块网格**渲染**成地板+墙+顶，而非地形 chunk。家具 item 层（palette/item_ref）沿用，落在房间网格内。

## 6. 资产
- **壁纸**：先复用现成 toy_wall 程序化纹理（点点墙纸，儿童房正搭），或新画 1-2 张壁纸材质；accent 墙后续。
- **地板**：木地板（现有 T_WOOD_FLOOR 纹理可复用为地板材质）/ 地毯后续。
- **门**：后续 prop；MVP 墙开口 + portal。
- 全部 CC0/程序化，无 IP。

## 7. 风险 / 待定（实现时定，眼验驱动）
- **相机框法**：整屋固定入镜 vs 房间内轻跟随。小房间倾向「整屋入镜、玩家在其中走」，最像动森。需真机眼验俯角/距离。
- **天花板**：建顶 mesh vs 暗色收顶。建顶更实但俯视可能挡视线/需半透明或前墙剖切；先眼验暗顶是否够。
- **前墙**：朝相机那面墙不建（好看进屋内），是动森常见做法；靠三面墙 + 收束相机成「屋」。
- **房间尺寸**：8×8 vs 10×10 vs 12×12，眼验取「不空旷又够摆」。
- **多房间/自定义房型**：MVP 一间固定；不做。

## 8. 粗 P 计划（审过后进 plan 模式细化）
- P1：RoomStage 节点（地板+三面墙+可选顶几何 + 壁纸/地板材质），离线 boot home_interior 眼验成「一间屋」。
- P2：渲染分支——进室内 build RoomStage、跳地形 chunk；出室内复位恢复地形；headless 不崩。
- P3：室内相机模式（`_update_camera` 加室内收束视角），进/出切换与复位；截图眼验框住整屋。
- P4：家具放置映射到房间网格（复用布置链，落点落在房间地板内）；实机眼验摆家具。
- P5：整合 portal（P2 复用）+ 家具起始包（P4 复用）+ 持久化；game-pilot 跑通「进屋→摆→出→再进还在」，截图眼验观感对齐参考。
- P6：全套回归 + merge --no-ff（prod 上线仍另议，存量迁移待定）。

## 9. 不做（明确排除）
- 多房间、自定义房型、串门进村民家。
- accent 墙 / 动态壁纸 / 地毯圈区 / 门 prop（先记为后续增强，非 MVP）。
- 家具经济（沿用 P4 起始包）。
