# 玩家间直接互动设计（player-interaction）

状态：**已拍板开工**（2026-07-11）——范围：打招呼 emote + 送爱心（纯计数）+ 玩家对话（ASR 文本中继）。

## 背景与现状

两台设备默认进同一世界（`world.gd` 硬编码 `api.get_world("default")`），多人可见性链路已完整：
presence（`actors_snapshot`/`actor_join` 带真实立绘）+ 位置流（`positions_relay` 插值副本）+
`actor_leave` 即时清理，均按场景定向广播（`WorldHub.broadcastScene`）。

但玩家之间**没有任何直接互动**：`_pick_npc`（world.gd）只遍历 `npcs`，`_remote_actors`
不可点选；对话系统（voice_input）只面向 NPC。现状 = 彼此可见的平行单机 + 共享沙盒
（地形/物品/造角色/演出间接共享）。

## 设计原则

1. **不做实时语音流。** 玩家对话走「本端 ASR → 文本中继 → 对端 TTS」（见 P5），
   不建 WebRTC/音频流通道——文本中继带宽近零、天然可插翻译层（老板目标：将来
   跨语言和其他世界的小朋友一起玩），也回避实时语音的技术重量与儿童安全问题。
2. **零文字、零层级菜单。** 点选 → 大图标盘 → 即时反馈，与造角色选项卡同一交互语言。
3. **只加不减。** 任何互动不得让对方失去东西（防大孩子薅小孩子）；赠与类默认不动
   钱包余额或服务端限额。
4. **薄叠在现有底座上。** 动作集复用 `behavior_executor.gd` 的 `ACTION_DUR`
   （wave/jump/spin/nod）；广播复用 `broadcastScene`；跑过去复用 approach 机制
   （`_check_approach`）；图标复用造角色图标生成管线。

## 分期

### P1 — 碰面打招呼（emote 互动）【推荐先做】

孩子 A 点孩子 B 的小人 → A 的角色跑过去 → 到位后弹 2×2 大图标表情盘
（👋挥手 / 跳一跳 / 转圈圈 / 点头）→ 选中后：

- A 端：本地角色播该动作 + 音效 + 头顶表情泡；
- 上行 `player_emote { targetPlayerId, action }`；
- 服务端：校验同世界同场景 → `broadcastScene` 转发（~30 行，无状态）；
- B 端（及同场景旁观者）：A 的远端副本播同款动作 + 表情泡；
- **自动回礼**：B 端自己的角色自动回一个 wave（孩子不操作也「有来有回」），
  同样广播回去。回礼不再触发回礼（防抖：同一对玩家 N 秒冷却）。

技术清单：
- 客户端：`_pick_remote_actor` 拾取扩展；approach 目标支持远端副本（目标会动，
  到位判定沿用 APPROACH_ARRIVE）；表情盘 UI（复用选项卡组件）；emote 播放
  （本地节点 + 远端副本共用 behavior_executor 动作路径）；表情泡（Label3D/贴片）。
- 服务端：`player_emote` 校验 + 场景定向转发；单测（转发、跨场景不转发、目标离场）。
- 资产：表情盘 4 图标（creation-icons 管线）。
- 测试：server 单测 + headless 冒烟（双 backend mock 互发 emote）。

### P2 — 送爱心【本期，已拍板方案 a】

表情盘加第 5 格「送❤」：送方播比心/挥手，收方头顶冒爱心特效。

爱心是**纯情感计数**，进集邮册新页「收到的爱心」，不动小红花钱包——零经济风险，
服务端只加一张邮票类目/计数。（方案 b「真转 1 朵小红花」已否决。）

### P5 — 玩家对话（ASR 文本中继）【本期，老板新增】

**不是**实时语音通话。链路：

1. 孩子 A 点孩子 B → 跑过去 → 进对话态（站桩构图对着 B 的副本，开麦，
   复用现有 VAD 断句 + 端侧 ASR）；
2. ASR 出文本后 **A 端先复述一遍**（用 A 自己的角色音色 TTS 播出）——孩子能听到
   自己被识别成了什么；
3. 同时上行 `player_speech { targetPlayerId, text, lang }` → 服务端校验同场景 →
   定向转发（含同场景旁观者，像现实里说话旁边人也听得见）；
4. B 端：A 的远端副本头顶气泡 + 同一文本用 A 的音色 TTS 播出。
   **A 端复述窗口 ≈ B 端收听窗口**（同文本同音色时长近似），节奏天然对齐；
5. 复述/播放期间本端麦静音（复用现有 TTS gating / interaction_fsm）。

**无会话锁**：这是「喊话」模型，不是「会话」模型。B 不需要进任何状态就能听见
（现实里有人跟你说话你也不用先按接听）；B 想回话就点 A 说回去。双方都点了对方
即自然形成双向对话。避免了双端状态机握手/打断对方当前活动的复杂度。

**玩家音色**：玩家档案分配 `voiceId`（预制音色池按 playerId 稳定哈希，同
greetingStyle 手法），随 presence 下发，对端据此合成。遵守「预制音色 = 运行期
音色」契约。

**翻译钩子（本期只留钩子不实现）**：`player_speech` 带 `lang` 字段；将来两端语言
不同时，服务端转发前过一层 LLM 翻译，B 端收到译文用 B 语言 TTS——跨语言共玩
只需加这一层。

### P3 — 同台演出（远期，本期不做）

点对方 → 「一起演戏」→ 双人 + NPC 进同一场 stage 演出，LLM 编剧把两个玩家都写进戏。
stage 已有玩家占角机制（`_stage_player_actor_id`），但多玩家占角/双端 tap 驱动
需要 StageDirector 扩展，成本高，待本期功能验证孩子真实用法后再评估。

## 明确不做

- 实时语音流/文字聊天 UI（原则 1；对话走 ASR 文本中继）。
- 玩家间物品转移/交易（原则 3；物品本就共享在世界里，摆下来对方就能拾取）。
- 碰撞/推挤等物理互动（远端副本是插值渲染，无物理体，且易引发负面玩法）。
- 本期不实现翻译，只留 lang 字段钩子。
